# プルリクエスト作成スキル

## 基本方針

- コミットメッセージ、PRタイトル、PR本文は日本語で書く。
- `main` ブランチ上で直接 PR 作成作業を進めない。必要なら `codex/` または既存方針に沿った作業ブランチを作る。
- まず Codex アプリの GitHub 連携を優先する。`gh` コマンドは、Codex アプリ連携が使えない、または権限不足で失敗した場合のフォールバックにする。
- 既存の未コミット変更を勝手に戻さない。ユーザーや他作業由来の変更が混ざっている場合は、対象ファイルだけを明示的にステージする。

## 手順

### 1. 現在状態を確認する

以下を確認する。

```bash
git status --short --branch
git branch --show-current
git log main..HEAD --oneline
git diff --stat
```

- 未コミット変更がある場合は、今回の依頼に関係するファイルだけをステージする。
- まだコミットがない場合は、日本語のコミットメッセージでコミットする。
- PR テンプレートがあれば読み込み、本文構成に反映する。

### 2. Codex アプリの GitHub 連携を優先する

Codex アプリ上で GitHub 連携ツールが使える場合は、ローカルの `gh` より先に使う。

推奨順:

1. GitHub 連携で対象リポジトリ、ベースブランチ、作業ブランチの状態を確認する。
2. 作業ブランチが GitHub 上に存在しない場合、GitHub 連携でブランチ作成を試す。
3. ローカルコミットをそのまま push できない場合は、GitHub 連携のファイル作成・更新 API で同等の変更を作業ブランチに反映できるか検討する。
4. 作業ブランチが GitHub 上に存在する状態になったら、GitHub 連携で PR を作成する。

Codex アプリ連携が以下のように失敗した場合は、結果をユーザーに明示し、フォールバックへ進む。

- `Resource not accessible by integration`
- `403`
- リポジトリやブランチへの書き込み権限不足
- GitHub 連携ツール自体が利用できない

### 3. フォールバックとして git / gh を使う

Codex アプリ連携で作成できない場合のみ、ローカル認証済みの git / gh を使う。

```bash
git push -u origin <branch>
gh pr create --title "<PRタイトル>" --body "<PR本文>"
```

`gh` が未インストール、または git push が認証エラーになる場合は、どこまで完了したか、ユーザーが次に実行すべきコマンド、PR本文案を報告する。

### 4. PR本文を作る

PR本文には最低限以下を含める。

```markdown
## 概要
変更の目的と要約を書く。

## 変更内容
- 主な変更点を書く。

## 確認
- 実行した確認を書く。
- 実行できなかった確認があれば理由を書く。

---
Generated with [Codex](https://Codex.ai/Codex)
```

### 5. 完了報告

PR 作成に成功したら、PR URL をユーザーに伝える。

作成できなかった場合は、失敗理由を短く明示し、次に必要な認証・操作を具体的に伝える。
